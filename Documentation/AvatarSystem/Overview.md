# Avatar System

## 1. Overview

The FlyChat avatar system renders per-sender circular avatars beside chat bubbles. It provides a three-tier resolution chain (remote image, default image, generated acronym), deterministic HSL coloring, built-in memory caching, and an optional custom cache delegate. The system is entirely internal to the package apart from the delegate protocols that host apps implement to supply URLs, caching, and visibility preferences.

Key files:

| File | Role |
|------|------|
| `Sources/FlyChat/Modules/Chat/View/FCLAvatarView.swift` | SwiftUI view + acronym generator + color generator |
| `Sources/FlyChat/Modules/Chat/Presenter/FCLAvatarImageCache.swift` | Built-in actor-isolated `NSCache` |
| `Sources/FlyChat/Core/Delegate/FCLAvatarDelegate.swift` | Public delegate & cache protocols |
| `Sources/FlyChat/Core/Delegate/FCLDelegateDefaults.swift` | Default values for avatar size & visibility |
| `Sources/FlyChat/Modules/Chat/Model/FCLImageSource.swift` | Sendable enum for fallback images |

## 2. Resolution Chain

When `FCLAvatarView` appears on screen its `.task(id: senderID)` modifier fires `loadAvatar()`. The function walks a strict fallback chain:

1. **Delegate URL** -- Call `avatarDelegate.avatarURL(for: senderID)`. If it returns a non-nil `URL`:
   a. **Check custom cache** -- If `avatarDelegate.cache` is non-nil, call `cacheDelegate.cachedImage(for: senderID)`. If `Data` is returned and converts to a valid `UIImage`, display it and stop.
   b. **Download** -- Perform `URLSession.shared.data(from: url)`. On success, display the decoded `UIImage`, then store the raw `Data` in the custom cache via `cacheDelegate.cacheImage(_:for:)`.
   c. If the download throws or yields non-image data, fall through.
2. **Default image** -- Read `avatarDelegate.defaultAvatarImage` (an `FCLImageSource?`). Resolve it:
   - `.name(String)` -- `UIImage(named:)` from the host app's asset catalog.
   - `.system(String)` -- `UIImage(systemName:)` from SF Symbols.
   If the image resolves, display it and stop.
3. **Acronym fallback** -- The view's default state is already the acronym circle (rendered by `acronymView`). If both prior steps fail, the acronym remains visible. The `loadFailed` flag is set to `true`.

Because the acronym view is the *initial* render state (not a deferred placeholder), the user never sees a blank frame -- the colorful initials circle is visible immediately while the async load runs.

## 3. Acronym Generation

Acronyms are generated by `FCLAvatarColorGenerator.initials(from:)`:

```swift
static func initials(from displayName: String) -> String {
    let components = displayName.split(separator: " ").map(String.init)
    guard !components.isEmpty, !components[0].isEmpty else { return "?" }
    let first = String(components[0].prefix(1)).uppercased()
    if components.count > 1, let last = components.last, !last.isEmpty {
        return first + String(last.prefix(1)).uppercased()
    }
    return first
}
```

### Rules

| Input | Output | Explanation |
|-------|--------|-------------|
| `"John Doe"` | `"JD"` | First letter of first + first letter of last component |
| `"Alice"` | `"A"` | Single component -- one letter |
| `""` | `"?"` | Empty or whitespace-only -- question mark sentinel |
| `"Mary Jane Watson"` | `"MW"` | First component + *last* component (middle names ignored) |

### Font sizing

The acronym `Text` uses a system bold font sized at **~40% of the avatar diameter**:

```swift
.font(.system(size: size * 0.4, weight: .bold))
```

With the default `avatarSize` of 40 pt this yields a 16 pt bold label. The text is always `.white` on top of the HSL-generated background circle.

## 4. HSL Color Generation

Each acronym circle gets a **deterministic** background color derived from the initials string. The pipeline is: initials -> djb2 hash -> hue -> fixed saturation & lightness -> HSL-to-HSB conversion -> SwiftUI `Color`.

### 4.1 djb2 Hash

```swift
static func djb2Hash(_ string: String) -> UInt {
    var hash: UInt = 5381
    for byte in string.utf8 {
        hash = ((hash &<< 5) &+ hash) &+ UInt(byte)
    }
    return hash
}
```

The classic djb2 algorithm by Daniel J. Bernstein. It iterates over the UTF-8 bytes of the input and accumulates `hash = hash * 33 + byte` using overflow operators (`&<<`, `&+`).

**Why djb2 and not `Swift.Hasher`?** `Hasher` uses per-process random seeding (introduced in Swift 4.2). This means `.hashValue` for the same string differs across app launches, so avatar colors would shuffle every time the user restarts the app. djb2 is fully deterministic -- the same initials always produce the same color, across launches, devices, and OS versions.

### 4.2 Hue Calculation

```swift
static func hue(for initials: String) -> Double {
    Double(djb2Hash(initials.uppercased()) % 360)
}
```

The hash is reduced modulo 360 to produce a hue in degrees. The input is uppercased first so `"jd"` and `"JD"` map to the same color.

### 4.3 HSL-to-HSB Conversion

SwiftUI's `Color(hue:saturation:brightness:)` uses the HSB (a.k.a. HSV) model, not HSL. The generator picks fixed HSL values (`s = 0.55`, `l = 0.45`) for a vibrant but not garish palette, then converts:

```swift
static func hslToHSB(h: Double, s: Double, l: Double) -> (h: Double, s: Double, b: Double) {
    let b = l + s * min(l, 1 - l)
    let sB = b == 0 ? 0 : 2 * (1 - l / b)
    return (h, sB, b)
}
```

The formulas:

```
brightness = lightness + saturation_HSL * min(lightness, 1 - lightness)
saturation_HSB = brightness == 0 ? 0 : 2 * (1 - lightness / brightness)
```

The resulting HSB tuple is fed to SwiftUI with hue normalized to 0...1:

```swift
Color(hue: hsb.h / 360, saturation: hsb.s, brightness: hsb.b)
```

## 5. Built-in Cache -- `FCLAvatarImageCache`

```swift
actor FCLAvatarImageCache {
    private let cache = NSCache<NSString, UIImage>()

    func image(for senderID: String) -> UIImage? {
        cache.object(forKey: senderID as NSString)
    }

    func store(_ image: UIImage, for senderID: String) {
        cache.setObject(image, forKey: senderID as NSString)
    }
}
```

Key characteristics:

- **Actor-isolated** -- all reads and writes are serialized, safe from any concurrency context without manual locking.
- **Keyed by sender ID** -- one cached image per unique sender, not per message.
- **Memory-only** -- backed by `NSCache`, which is an in-memory LRU cache. No disk persistence.
- **Auto-eviction** -- `NSCache` automatically purges entries under memory pressure; no manual `countLimit` or `totalCostLimit` is set, so the OS governs eviction.
- **UIKit-gated** -- the entire file is wrapped in `#if canImport(UIKit)`.

> **Note:** The current `FCLAvatarView.loadAvatar()` method uses the *custom* cache delegate (`avatarDelegate.cache`) rather than the built-in `FCLAvatarImageCache`. The built-in cache actor is available as an internal utility for future use or for presenters to leverage directly.

## 6. Custom Cache -- `FCLAvatarCacheDelegate`

```swift
public protocol FCLAvatarCacheDelegate: AnyObject, Sendable {
    func cachedImage(for senderID: String) async -> Data?
    func cacheImage(_ data: Data, for senderID: String) async
}
```

### Design Decisions

| Decision | Rationale |
|----------|-----------|
| **`Data`-based, not `UIImage`** | `Data` is `Sendable`; `UIImage` is not. This keeps the protocol safe to call from any isolation domain without `@unchecked Sendable` wrappers. It also allows disk-backed caches to store raw bytes without re-encoding. |
| **`Sendable` conformance** | Required so the cache can be stored and called across actor boundaries (the delegate is accessed from `@MainActor` view code and could be implemented on a background actor). |
| **Not `@MainActor`** | Cache operations are I/O work that should not block the main thread. The protocol's `async` methods let implementations run on any executor. |
| **`AnyObject` (class-only)** | Ensures reference semantics so the same cache instance is shared, not copied. |

### Full Conformance Example

```swift
import Foundation

final class DiskAvatarCache: FCLAvatarCacheDelegate, @unchecked Sendable {
    private let directory: URL
    private let queue = DispatchQueue(label: "com.app.avatar-cache")

    init() {
        let caches = FileManager.default.urls(for: .cachesDirectory, in: .userDomainMask)[0]
        self.directory = caches.appendingPathComponent("avatars", isDirectory: true)
        try? FileManager.default.createDirectory(at: directory, withIntermediateDirectories: true)
    }

    func cachedImage(for senderID: String) async -> Data? {
        let fileURL = directory.appendingPathComponent(senderID.addingPercentEncoding(withAllowedCharacters: .alphanumerics) ?? senderID)
        return try? Data(contentsOf: fileURL)
    }

    func cacheImage(_ data: Data, for senderID: String) async {
        let fileURL = directory.appendingPathComponent(senderID.addingPercentEncoding(withAllowedCharacters: .alphanumerics) ?? senderID)
        try? data.write(to: fileURL, options: .atomic)
    }
}
```

Wire it up by returning the instance from your `FCLAvatarDelegate`:

```swift
var cache: (any FCLAvatarCacheDelegate)? { diskCache }
```

## 7. `FCLImageSource`

```swift
public enum FCLImageSource: Sendable, Hashable {
    case name(String)
    case system(String)
}
```

| Case | Resolution |
|------|-----------|
| `.name(String)` | `UIImage(named:)` -- loads from the host app's asset catalog |
| `.system(String)` | `UIImage(systemName:)` -- loads an SF Symbol |

### Why no `.image(UIImage)` case?

`UIImage` does not conform to `Sendable`. Including it in the enum would break the enum's `Sendable` conformance, which is required because `FCLImageSource` is stored on the `@MainActor`-isolated `FCLAvatarDelegate` protocol and read during async avatar loading. Keeping the enum purely `String`-based preserves full `Sendable` + `Hashable` safety with zero escape hatches.

## 8. Visibility Rules

### Defaults (from `FCLDelegateDefaults.swift`)

```swift
enum FCLAvatarDefaults {
    static let avatarSize: CGFloat = 40
    static let showOutgoingAvatar = false
    static let showIncomingAvatar = true
}
```

| Property | Default | Meaning |
|----------|---------|---------|
| `showIncomingAvatar` | `true` | Incoming messages show an avatar |
| `showOutgoingAvatar` | `false` | Outgoing messages do not show an avatar |
| `avatarSize` | 40 pt | Diameter of the avatar circle |

Host apps override these by implementing the corresponding properties on their `FCLAvatarDelegate` conformance.

### Group Positioning

Within a consecutive group of messages from the same sender, only the **last message in the group** displays the actual avatar. All earlier messages in that group render an invisible spacer of the same size to maintain consistent horizontal alignment.

The grouping logic lives in `FCLChatPresenter`:

```swift
public func isLastInGroup(for message: FCLChatMessage) -> Bool {
    guard let index = messages.firstIndex(where: { $0.id == message.id }) else { return true }
    return index == messages.count - 1 || messages[index + 1].direction != message.direction
}
```

A message is "last in group" when:
- It is the final message in the entire list, **or**
- The next message has a different `direction` (i.e., the sender switches).

### Layout Alignment

The `HStack` containing bubble + avatar uses `.bottom` alignment:

```swift
HStack(alignment: .bottom, spacing: showAvatar ? 8 : 0) { ... }
```

This bottom-aligns the avatar circle with the bottom edge of the bubble, mimicking the Telegram/iMessage visual pattern where the avatar sits at the baseline of the last bubble in a group.

When `showAvatar` is `false` for a direction, spacing collapses to 0 and no avatar column is rendered at all.

## 9. `FCLAvatarView` -- Internal Component

`FCLAvatarView` is an `internal` SwiftUI view (not exposed to host apps). It is UIKit-gated with `#if canImport(UIKit)`.

### Properties

```swift
struct FCLAvatarView: View {
    let senderID: String
    let displayName: String
    let size: CGFloat
    let delegate: (any FCLAvatarDelegate)?

    @State private var loadedImage: UIImage?
    @State private var loadFailed = false
    // ...
}
```

### Lifecycle: `.task(id:)`

```swift
.task(id: senderID) {
    await loadAvatar()
}
```

The `id: senderID` parameter is critical. It means:

- The task fires when the view first appears **and** whenever `senderID` changes (e.g., if SwiftUI recycles the view for a different cell).
- If `senderID` changes, SwiftUI **cancels** the previous task before starting a new one. This provides automatic cancellation during fast scrolling -- if a cell is reused before its download completes, the in-flight `URLSession` request is cancelled via Swift Concurrency's cooperative cancellation.

### Rendering States

| State | What is shown |
|-------|--------------|
| Initial (before/during load) | Acronym circle with HSL background |
| Load succeeded | Downloaded/cached `UIImage`, clipped to circle |
| Load failed | Acronym circle remains (no error indicator) |

The view is always clipped to a circle:

```swift
.frame(width: size, height: size)
.clipShape(Circle())
```
